## 1. 專題摘要

本專題開發一套名為 **Wi-Care** 的智慧長照監控系統，針對高齡化社會下長照機構與居家照護的需求，提供一種**非接觸式、隱私保護**的即時監控解決方案。

系統核心採用 **ESP32-S3 微控制器**搭配 **ESPectre v2.7.0 韌體**，透過 Wi-Fi 通道狀態資訊（CSI, Channel State Information）技術，在完全不使用攝影機、麥克風或穿戴裝置的前提下，實現：

- 🔍 **即時跌倒偵測**（沿用 ESPectre MVS 核心，文獻 F1 > 96%）
- 📍 **Wi-Fi 室內定位**（三角定位法）
- 📊 **人體活動辨識**（6 級分類）
- 📱 **LINE Notify 即時推播**
- 🖥️ **React 即時監控儀表板**

**關鍵字**：Wi-Fi CSI、長照監控、跌倒偵測、室內定位、IoT、React

## 2. 研究動機與目的

### 2.1 研究背景

台灣已進入**超高齡社會**，65 歲以上人口比例持續上升。根據衛福部統計，高齡者跌倒是造成意外死亡的第二大原因，且跌倒後若未能即時發現與救援，將大幅提升致死率與後遺症發生率。

傳統監控手段面臨兩大矛盾：

| 問題 | 傳統方案 | 缺點 |
|------|---------|------|
| 跌倒偵測 | 攝影機監控 | 嚴重侵害隱私 |
| 活動偵測 | 穿戴裝置（手環、感測器） | 長者依從性差、需充電維護 |
| 位置追蹤 | GPS 或 RFID | 室內精度不足、有穿戴負擔 |

### 2.2 研究目的

本專題旨在解決上述矛盾，達成以下目標：

1. **保護隱私**：不使用任何影像或聲音設備
2. **免穿戴**：長者無需配戴任何感測器
3. **即時性**：跌倒偵測延遲 ≤ 1 秒
4. **低成本**：硬體成本低於傳統方案
5. **易部署**：利用現有 Wi-Fi 基礎設施

### 2.3 技術創新點

Wi-Care 的技術核心是 **Wi-Fi CSI 感測技術**，其原理為：

> 人體在 Wi-Fi 訊號傳播路徑中移動時，會改變多路徑效應，導致各子載波的振幅與相位產生變化。通過分析這種變化，可以推斷環境中的人體行為。

此方法被稱為「**通道狀態資訊感測**（CSI Sensing）」，近年來已成為無線感知領域的研究熱點。

### 4.2 ESPectre v2.7.0 韌體功能

ESPectre 是專為 CSI 感測設計的開源韌體，主要功能：

- **CSI 採集**：被動監聽 Wi-Fi 封包，萃取通道狀態資訊
- **NBVI 計算**：Normalized Bandwidth Variance of Subcarriers（子載波帶寬方差）
- **Movement Score 輸出**：將 NBVI 正規化為 0-100+ 的移動分數
- **BLE 傳輸**：透過 GATT Notify 特性即時推送資料

## 5. 後端核心模組

### 5.1 `core_bridge.py` 主程式

後端採用 **Python 3.12** 撰寫，主要模組說明：

#### SharedState（共用狀態管理）

```python
class SharedState:
    """執行緒安全的共用狀態，使用 threading.Lock 保護"""
```

- 解析格式：`mvmt:X.XX`（X.XX 為浮點數）

- 使用 `bleak` 函式庫（非同步 BLE 客戶端）

#### WebSocket Server（前端通訊）

## 7. 跌倒偵測演算法

### 7.1 演算法概述

系統提供兩種跌倒偵測方式，可於設定頁面切換：

| 模式 | 全名 | 實作狀態 | 特點 |
|------|------|---------|------|
| MVS | Moving Variance Segmentation | ✅ 已實作 | 傳統統計方法，免訓練，自適應閾值 |
| ML 閘門 | 規則式信心閘門 | ✅ 已實作 | 固定比例閘門（免校正、即時啟動） |
| ESPectre 原生 | 韌體內置偵測 | ✅ 已實作 | 硬體直接判斷，最低延遲 |

> **誠實聲明**：本系統目前的「ML 模式」為**規則式信心閘門**（movement ≥ threshold 且
> 移動分數 ≥ 100% 時判定），**並非已訓練的神經網路**。真正的 MLP 神經網路（含資料蒐集、
> 標註、訓練、模型嵌入）列為**未來工作**（見 §13.3），本文不宣稱其效能數據。

### 7.2 MVS 演算法詳解

**步驟一：建立移動分數緩衝區**

```
buffer = deque(maxlen=300)  # 儲存最近 300 秒的分數
```

**步驟二：計算動態閾值**

```
P95 = percentile(buffer, 95)      # 第 95 百分位
threshold = P95 × 1.1             # 自適應閾值
```

**步驟三：上升邊緣偵測**

```
if (current_score > threshold) AND (previous_score ≤ threshold):
    is_falling = True
    start_cooldown(60s)  # 60 秒冷卻期
```

**步驟四：閾值模式選擇**

```
auto:   threshold = P95 × 1.1      # 自動校正（推薦）
min:    threshold = P100           # 最高分數（最寬鬆）
manual: threshold = user_input     # 手動設定
```

### 7.3 ML 模式（規則式信心閘門）與未來的神經網路設計

**目前實作（規則式閘門）**：為了在尚未蒐集訓練資料的情況下提供「免校正、即時啟動」的
替代選項，ML 模式採用簡單的規則閘門：

```
if (movement_score ≥ 100%) AND (movement ≥ threshold):
    → 判定為跌倒風險
else:
    → 正常
```

此法**無模型、無訓練**，僅為展示用的高靈敏度替代方案。

## 8. Wi-Fi 室內定位

### 8.1 定位原理

Wi-Fi 室內定位採用**對數距離路徑損耗模型**（Log-Distance Path Loss Model）：

**公式：**

$$d_i = 10^{\frac{A - RSSI_i}{10 \times n}}$$

| 符號 | 說明 | 預設值 |
|------|------|-------|
| $d_i$ | 設備到第 i 個 AP 的距離（公尺） | - |
| $A$ | 距離 1 公尺時的參考訊號強度（dBm） | -45 dBm |
| $RSSI_i$ | 第 i 個 AP 的接收訊號強度（dBm） | 實測值 |
| $n$ | 路徑損耗指數（環境衰減係數） | 2.7 |

### 8.2 多邊定位演算法

取得各 AP 的估算距離後，使用**最小平方法（Least Squares）**求解位置：

$$\text{minimize} \sum_{i=1}^{N} \left( \|\mathbf{p} - \mathbf{AP}_i\| - d_i \right)^2$$

其中 $\mathbf{p} = (x, y)$ 為待求位置。

### 8.3 定位流程

```
第 1-6 回合 Wi-Fi 掃描（間隔 0.7 秒）
    ↓
對每個 AP 的 RSSI 取平均值（減少雜訊）
    ↓
RSSI → 距離轉換（對數距離模型）
    ↓
最小平方法多邊定位
    ↓
輸出 (X, Y) 座標，更新前端地圖
```

### 8.4 環境校準參數

| 環境類型 | 路徑損耗指數 n | 說明 |
|---------|--------------|------|
| 無障礙空間（戶外） | 2.0 | 視線直達 |
| 一般室內 | 2.7 | **預設值** |
| 多牆壁室內 | 3.0 | 隔間多 |
| 高障礙環境 | 3.0 ~ 4.0 | 鋼筋混凝土 |

## 9. 使用者功能說明

### 9.1 登入與帳號管理

**一般帳號登入流程：**
1. 選擇使用者身份（醫護人員 / 家屬 / 管理者）
2. 輸入帳號與密碼
3. 點選「登入系統」

**Google 帳號登入流程：**
1. 點選「使用 Google 帳號登入」
2. 選擇 Google 帳號並授權
3. 首次登入自動建立帳號，進入系統

### 9.2 即時監控儀表板

儀表板提供以下即時資訊：

- **移動分數折線圖**：以 10 Hz 更新頻率顯示 CSI 移動分數歷史趨勢
- **活動狀態指示**：根據分數顯示 6 級活動狀態
- **跌倒警報**：偵測到跌倒時彈出紅色警報通知
- **室內位置地圖**：顯示被照護者在房間內的估算位置

**6 級活動狀態分類：**

| 分數範圍 | 活動等級 | 說明 |
|---------|---------|------|
| 0 - 10 | 睡眠 | 靜止不動 |
| 10 - 25 | 靜坐 | 輕微呼吸動作 |
| 25 - 45 | 輕微活動 | 轉身、手部動作 |
| 45 - 65 | 行走 | 室內步行 |
| 65 - 85 | 激烈活動 | 快速移動 |
| 85+ | 跌倒風險 | 異常尖峰，觸發警報 |

### 9.3 警報推播系統

跌倒事件偵測後，系統執行以下動作：

1. **前端警報**：即時顯示紅色警報視窗，包含時間、位置資訊
2. **LINE Notify 推播**：傳送通知至照護人員 LINE 帳號
3. **事件記錄**：將警報寫入警報歷史，供後續查閱
4. **冷卻期控制**：60 秒內不重複推播，避免騷擾

LINE Notify 訊息格式：
```
🚨 跌倒偵測警報！
時間：2026-05-25 14:30:00
位置：(3.2m, 4.1m)
移動分數：123.5
請立即前往確認！
```

### 9.4 健康記錄管理

| 記錄類型 | 欄位 | 說明 |
|---------|------|------|
| 日常健康 | 血壓（收縮壓/舒張壓）、血氧飽和度 | 每日記錄 |
| 例行健檢 | 體重、血糖、尿檢、便檢 | 定期健檢 |
| 趨勢分析 | 折線圖 | 長期健康趨勢 |

### 9.5 系統設定

| 設定項目 | 選項 | 說明 |
|---------|------|------|
| 演算法模式 | MVS / ML | 跌倒偵測演算法選擇 |
| 閾值模式 | Auto / Min / Manual | 觸發閾值設定方式 |
| 靈敏度 | 0-100% | 偵測靈敏度調整 |
| Hampel 濾波 | 開/關 | 離群值過濾 |
| 低通濾波 | 開/關 | 高頻雜訊過濾 |
| LINE Token | 文字輸入 | LINE Notify 推播設定 |

### 11.2 安裝步驟

**步驟 1：取得原始碼**
```bash
git clone https://github.com/VincenTddos/CSI-front.git
cd CSI-front
```

```bash
npm install
```

```bash
# 模擬模式（無需硬體）

python core_bridge.py --ble

# Serial USB 模式
python core_bridge.py --port COM7   # Windows
python core_bridge.py --port /dev/ttyUSB0  # Linux
```

**終端機 2：啟動前端**
```bash
npm run dev
```

**開啟瀏覽器**：`http://localhost:3000`

## 13. 結論與未來展望

### 13.1 成果總結

本專題成功開發完成 Wi-Care 智慧長照監控系統，達成預定目標：

✅ **非接觸式感測**：完全不需要攝影機、麥克風或穿戴裝置  
✅ **即時跌倒偵測**：沿用 ESPectre MVS 偵測核心（文獻 F1 > 96%），本系統偵測延遲 < 200ms  
✅ **室內定位**：以對數距離路徑損耗模型估算房間內區域位置  
✅ **即時推播**：LINE Notify 整合，照護人員即時接收警報  
✅ **完整管理系統**：住民資料、健康記錄、多角色帳號管理  
✅ **現代化 UI**：React 19 響應式設計，支援 Google 帳號登入  

### 13.2 技術貢獻

1. **低成本方案**：整體硬體成本（ESP32-S3）約 300-500 元新台幣
2. **隱私保護設計**：CSI 訊號不包含任何可識別個人資訊
3. **模組化架構**：易於擴充，可新增多房間、多設備支援
4. **開源專案**：程式碼公開於 GitHub，可供學術研究與商業應用

### 13.3 未來展望

| 改進方向 | 說明 | 預計效益 |
|---------|------|---------|
| 多房間部署 | 支援多個 ESP32 協同感測 | 覆蓋整個長照機構 |
| 深度學習模型優化 | 使用更大規模訓練資料 | F1 提升至 98%+ |
| 邊緣運算 | 在 ESP32 端直接執行 AI | 降低延遲至 < 50ms |
| 姿態估計 | 區分跌倒類型（前/後/側） | 提供更精準的緊急救護資訊 |
| 雲端整合 | 資料上傳雲端，多設備同步 | 遠端監控、大數據分析 |
| 睡眠品質分析 | 透過夜間 CSI 分析睡眠週期 | 擴展健康監控範疇 |
| 多語言支援 | 英文 / 日文介面 | 擴大使用族群 |

### 13.4 心得感想

透過本專題的開發，團隊深入學習了：
- Wi-Fi CSI 感測原理與 ESP32 韌體開發
- Python AsyncIO 非同步程式設計
- React 19 + TypeScript 現代前端開發
- WebSocket 即時通訊架構設計
- IoT 系統的整合與部署

最大的挑戰在於 CSI 訊號的不穩定性——環境中任何移動（包含寵物、窗簾搖動）都可能造成誤報。透過多層濾波設計與動態閾值機制，成功將誤報率控制在可接受範圍內。
